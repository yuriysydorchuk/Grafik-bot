# Telegram-бот (`bot/`)

Уся логіка Telegram-бота системи **Grafik-bot**. Бот і REST API працюють в одному Node-процесі (див. [`../README.md`](../README.md) та [`/CLAUDE.md`](../../../../CLAUDE.md)).

## Як запускається

- `instance.ts` створює єдиний екземпляр `new Telegraf(TELEGRAM_BOT_TOKEN)` (експорт `bot`). Без токена — кидає помилку на старті.
- `bot/index.ts` реєструє всі обробники (middleware, команди, `hears`, `action`, `on`) — імпорт цього модуля «підключає» їх до `bot`.
- `src/index.ts` (точка входу процесу) викликає `bot.launch()` **без `await`**: у Telegraf цей promise резолвиться лише коли polling зупиняється, тому його не можна чекати в лінію зі стартом сервера. Перед launch викликається `loadStates()` (відновлення діалогів).
- Зупинка: `bot.stop("SIGINT"|"SIGTERM")` у graceful-shutdown.

## Long-polling і правило одного інстансу

Бот працює в режимі **long-polling** (без webhook) — сам опитує Telegram. **Критично: лише один інстанс на токен.** Якщо одночасно працюють локальний запуск і прод (або два pm2-процеси), Telegram повертає `409 Conflict` і повідомлення «стрибають» між інстансами.

Практично:
- не запускай `dev` локально, поки прод-бот живий на тому ж токені (для локальної розробки потрібен окремий тестовий бот/токен);
- у pm2 має бути рівно один процес `grafik-bot`.

## Структура

```
bot/
├── instance.ts   єдиний екземпляр Telegraf (експорт `bot`)
├── index.ts      усі обробники (~3200 рядків): middleware, команди, hears, actions, on(text/photo/document)
├── menus.ts      reply-keyboard меню за ролями (adminMenu, workerMenu, driverMenu, headDriverMenu, managementMenu)
├── views.ts      рендер екранів (графік водія, тиждень, слоти тощо)
├── i18n.ts       багатомовність: t/tb/bhears, asLang, LANGS (5 мов працівника; uk/en для офісу/водія)
├── roles.ts      isAdmin / getAdmin / getWorker / getDriver (визначення ролі за telegramId)
├── state.ts      діалоговий стан: in-memory Map + write-through у БД (user_states)
├── notify.ts     вихідні сповіщення (графіки, відсутності, бонуси, розсилки, сповіщення ролям)
├── chat.ts       трекінг повідомлень бота (для масового очищення < 48 год) + pruning
├── display.ts    хелпери форматування (DAY_UK, SHIFT_SHORT, splitMessage, escapeHtml)
├── time.ts       час/зміни у Europe/Warsaw (+ time.test.ts)
└── README.md     цей файл
```

## Основні обробники / сцени / команди

Бот не використовує Telegraf Scenes — багатокрокові діалоги реалізовані через **власний стан** (`state.ts`).

- **Команди** (`bot.command`): `/adminsetup` (первинна привʼязка головного адміна), `/getid` (показати свій Telegram ID), `/invite`.
- **`bot.start`** — обробка deep-links `?start=...`: реєстрація працівника за кодом/посиланням фабрики, привʼязка адміна/водія за invite-кодом, вибір мови.
- **Middleware** (`bot.use`) — визначення ролі та контексту перед обробкою.
- **`bot.hears` (~53)** — натискання кнопок reply-keyboard меню. Кнопки двомовні: `bhears([...])` матчить варіанти різними мовами. Меню залежить від ролі (`menus.ts`).
- **`bot.action` (~32)** — inline-кнопки (callback queries): підтвердження відсутності (прийняти/заміна/відхилити), вибір мови (`olang:*`), редагування графіку, дошки замовлень тощо.
- **`bot.on("text"|"photo"|"document")`** — введення в межах активного діалогу: текстові кроки (`state.action`), фото місячного звіту, Excel-імпорт графіку.

**Багатокрокові діалоги:** `setState(id, action, data)` зберігає крок (ключ = Telegram ID), `getState`/`clearState` — читання/скидання. Стан кешується в памʼяті та дзеркалиться в таблицю `user_states`, тож **діалоги переживають перезапуск/деплой** (відновлюються через `loadStates()` на старті).

**Ролі бота:** працівник (графік, зголосити відсутність, звіт), водій (явка, поїздки, недовоз), головний водій (призначення водіїв, повний графік), офіс/адмін (керування — переважно у веб-панелі).

## Звʼязок з API / services / БД

- **БД:** напряму через `@workspace/db` (Drizzle) — ті самі таблиці, що й веб-панель.
- **services:** `index.ts` викликає `../services/sheets` (доступність), `../services/scheduleGenerator` (генерація), `../services/drive` (Excel/Drive); подекуди динамічний `import()` (`scheduler`, `email`) щоб уникнути циклів.
- **notify.ts** — централізовані вихідні повідомлення через `bot.telegram.sendMessage` (графіки працівникам/водіям, сповіщення про відсутність, бонуси, розсилки, `notifyRoles` для веб-сповіщень).
- **Веб ↔ бот:** спільні через БД. Напр., логін у панель надсилає код через бота (`notify.sendLoginCode`); затвердження графіку у веб тригерить розсилку ботом.

## Cron / scheduler-взаємодія

Планувальник `services/scheduler.ts` **імпортує `bot`** і шле повідомлення через нього. Завдання (node-cron, Europe/Warsaw):

- **Щотижневе нагадування** — неділя о `reminderHour`:00 (типово 18:00): кому не заповнив доступність на наступний тиждень + зведення адмінам.
- **Пресмінні сповіщення** — кожні 15 хв: за ~2 год до старту зміни нагадує працівникам і водієві (дедуп через `sentToday`).
- **Скидання дедупу** — опівночі.
- **Housekeeping** — 04:00: прибирання трекінгу повідомлень (`chat.pruneOldMessageRows`) і старих сповіщень.

Зміна години нагадування (`setReminderHour`) перезапускає cron-завдання.

## Ризики при деплої через pm2

- **Подвійний polling.** Найчастіша проблема: старий процес не зупинився, а новий уже стартував → `409 Conflict`. Перед стартом переконайся, що живий рівно один `grafik-bot` (`pm2 list`); використовуй `pm2 restart grafik-bot`, не «другий start».
- **Локальний + прод на одному токені** — той самий конфлікт. Для локального тесту — окремий тестовий бот.
- **Збирати перед рестартом.** pm2 запускає `dist/index.mjs`; після зміни коду спершу `pnpm --filter @workspace/api-server run build`, потім `pm2 restart`.
- **Env через pm2.** Токен/`DATABASE_URL` беруться з `--env-file=./.env` (див. `ecosystem.config.cjs`). Якщо `.env` не там, де очікує `cwd`, бот не підніметься.
- **Тимчасова зупинка polling під час рестарту** — невеликий «провал» вхідних повідомлень, поки процес перезапускається; зазвичай Telegram добирає їх після підняття. Незавершені діалоги не губляться (стан у `user_states`).
- **Один процес = усе разом.** Падіння бекенду/бота кладе і веб-панель. pm2 `autorestart` піднімає назад (max_restarts=30, restart_delay=3000).
- **Webhook не використовується** — нічого «прокидати» ззовні не треба; але переконайся, що для цього токена не залишився старий webhook (інакше polling не отримуватиме апдейти).
