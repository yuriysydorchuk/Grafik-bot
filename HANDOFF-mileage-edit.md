# HANDOFF — редагування пробігу (одометра)

> Одноразова записка сесії 2026-07-04. Довговічні факти вже перенесені: маршрут — у `PROJECT_MAP.md` (розділ «Водії»). Після ознайомлення — перемістити в `docs/archive/`.

## Що зроблено (задеплоєно на прод, коміт `6e1534b`)

Адмін і головний водій можуть виправляти пробіг змін водіїв у веб-панелі; водії — самі в боті протягом 24 годин.

### API (`artifacts/api-server/src/routes/admin-api.ts`)
- `GET /mileage` тепер віддає `id` кожного workday-рядка (потрібен для PATCH).
- Новий `PATCH /driver-workdays/:id` під `DRIVER_RW` (= `editData` АБО `assignDrivers`) — без часових обмежень.
  Валідація: ціле 0–3 000 000; кінцевий ≥ початкового; кінцевий пробіг **відкритої** зміни редагувати не можна (400).

### Веб (`artifacts/web/src/pages/Mileage.tsx`)
- Олівець у рядку таблиці «Звіт по пробігу» → інлайн-редагування початкового/кінцевого, ✓/✕, тост, invalidate `["mileage"]`.
- Кнопка видима лише при `can(me,"editData") || can(me,"assignDrivers")`.
- Переклади EN + RU у `web/src/lib/i18n.tsx` (сторінка водійська → RU додано).

### Бот (`artifacts/api-server/src/bot/index.ts` + `bot/i18n.ts`)
- Після «✅ Зміну розпочато!» / «🏁 Зміну закінчено!» бот шле **окреме** повідомлення
  «Помилилися з пробігом? Виправити можна протягом 24 годин:» з inline-кнопками
  (`wdfix:start:<id>`, для закритої зміни ще `wdfix:end:<id>`). Окреме — бо reply-меню
  (`driverMenuFor`) та inline-клавіатура не поєднуються в одному повідомленні.
- Вікно 24 год рахується від моменту запису показника: старт — від `startedAt`, кінець — від `endedAt`
  (константа `WORKDAY_FIX_WINDOW_MS`). Пізніше — «виправити може лише адміністратор».
- Новий стан діалогу `workday:fix_km` (обробник у `bot.on("text")` після `workday:end_km`):
  та сама валідація числа + перехресна (старт ≤ кінець проти актуального запису з БД), підтвердження з перерахованими км.
- `db.insert(driverWorkdaysTable)` при старті зміни тепер з `.returning()` (потрібен id для кнопки).
- i18n: усі нові рядки додані в `BOT_EN` і `BOT_RU` (водійський досвід — uk/en/ru).

## Деплой / перевірка
- `pnpm run typecheck`, збірка web+api, `node --test` — чисто.
- Прод: `git pull` + `deploy/build.sh` виконані; `healthz` → `{"status":"ok","db":"ok","bot":"up"}`, полінг стартував без 409.
- Смоук вручну (не робився): у боті стартувати зміну з помилковим числом → «✏️ Виправити пробіг»; у вебі — олівець на «Звіті по пробігу» під owner і під головним водієм.

## Нюанси / можливі follow-up
- Підказка з inline-кнопкою шлеться після **кожного** старту/кінця зміни (2-ге повідомлення). Якщо здасться шумним — можна прибрати текст і лишити кнопку, або показувати рідше.
- 24h-перевірка робиться при натисканні кнопки; сам ввід числа після цього не пере-перевіряє вікно (не критично: стан живе до вводу/скасування).
- У веб-редагуванні порожній кінцевий пробіг для закритої зміни піде як `0` і впаде на серверній валідації (повідомлення «кінцевий < початкового») — за бажання додати клієнтську перевірку.
- Кінцевий пробіг відкритої зміни свідомо не редагується ніде (закривати зміну має бот-флоу «🏁 Закінчити зміну»).
