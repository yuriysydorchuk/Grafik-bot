Коміт `62ec0d7`. **Три міграції** (накочені на прод): `2026-07-11-worker-invite-token.sql`, `2026-07-11-admin-token-version.sql`, `2026-07-11-admin-sessions.sql`. Смоук пройдено (`healthz` ok, бот up). Джерело задачі — аудит уразливостей на початку сесії.

## Що зроблено

1. **HIGH: захоплення непривʼязаних профілів працівників закрито.** `worker_code` був послідовним (`00001…`) і водночас секретом deep-link привʼязки (`?start=<code>`) → перебірний. Тепер привʼязка — окремим криптотокеном `workers.invite_code` через `?start=emp<code>` (одноразовий, обнуляється при claim). `worker_code` лишився **лише публічним** id для показу. Гілка привʼязки за голим `worker_code` у `bot/index.ts` **видалена**.
2. **Усі invite/2FA-коди — crypto, не `Math.random`.** Новий `lib/invite.ts` `randomInviteCode` (base32, ≥12 симв.). Переведено: коди водіїв (створення + regen в `admin-api.ts`, `genDriverCode` в боті), адмінів (`uniqueAdminCode`), 2FA-код входу (`crypto.randomInt`).
3. **Відкликання сесій — 2 рівні.** `admins.token_version` вшивається в токен (`tv`), `authRequired` звіряє щозапиту. Зміна пароля (`web_login:password`, `reset-web`) і «вийти скрізь» → інкремент версії. Сесії ще й **відстежувані**: токен несе `sid` (рядок `admin_sessions`); `authRequired` перевіряє наявність/`revoked_at` → блокування **окремої** сесії. `logout` = ревок лише поточної.
4. **Журнал входів + сторінка `/security`** (лише `is_main`, як `/admins`). `login_events` пише success/bad_password/bad_2fa/no_telegram/logout; `admin_sessions` тримає активні. UI (`web/src/pages/Security.tsx`): активні сесії (пристрій/IP/гео/час) з кнопками 🔒 блокувати + «вийти скрізь», і журнал. Бекенд — `routes/security.ts` (4 ендпоінти).
5. **Гео входів** — `lib/clientInfo.ts`: парсер User-Agent («Chrome на macOS») + best-effort `ipwho.is` (keyless, fire-and-forget, гейт `GEOIP_ENABLED`, приватні/loopback IP пропускаються).

## Нюанси / підводні камені

- **Один раз розлогінить усіх при цьому деплої** — старі токени без `sid` не проходять новий `authRequired`. Це очікувано (перший вхід і запускає трекінг сесій). Уже відбулось.
- **Старі посилання `?start=<worker_code>` (напр. `?start=00042`) більше не привʼязують** — навмисно. Офіс має **перевидати** посилання непривʼязаним працівникам (кнопка в Workers / «Запросити всіх» → генерують `emp`-токени). Привʼязані не зачеплені.
- **Гео локально завжди порожнє** — на localhost IP = `::1` (loopback), геолокації нема. UI показує «локальна мережа». На проді IP публічний (Caddy `X-Forwarded-For` + `trust proxy=1`).
- **`last_seen_at`** оновлюється з тротлінгом (раз/5хв), щоб не писати в БД щозапиту.
- **`worker_code` досі генерується** (`genWorkerCode` / `nextWorkerCode`) — це нормально, він публічний id, не секрет.
- `token_version` і per-session `revoked_at` — **два незалежні** механізми; `authRequired` перевіряє обидва. Не плутати: пароль/«скрізь» = `token_version`; одна сесія = `revoked_at`.

## Перевірити живцем (на проді)

- [ ] Відкрити **Безпека / Сесії** (головним адміном) → видно свою активну сесію; **IP публічний** (не `127.0.0.1`) і місто/країна заповнені.
- [ ] Спробувати невірний пароль → у журналі зʼявляється `Невірний пароль` з IP/пристроєм.
- [ ] «Заблокувати» другу сесію (з іншого браузера/пристрою) → та сесія одразу розлогінюється (наступний запит 401).
- [ ] «Вийти на всіх пристроях» для себе → усі сесії завершено, треба увійти заново.
- [ ] Новому працівнику: кнопка «посилання» в Workers дає `?start=emp…`; перехід привʼязує; повторний перехід тим самим посиланням → «недійсне/застаріле».
- [ ] Створити водія → invite-лінк тепер `?start=drv<крипто-код>` (не 5 цифр).
- [ ] Якщо гео не пішло на проді — звірити, що `req.ip` публічний; за потреби підкрутити `trust proxy` під реальну к-сть проксі.
