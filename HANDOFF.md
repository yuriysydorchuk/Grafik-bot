## Короткий контекст

**Grafik-bot** — система планування персоналу для кадрової агенції (бренд Euro Support): Telegram-бот (працівники/водії/офіс, багатомовний) + веб-панель адміністратора (укр/англ) + REST API + PostgreSQL. Монорепо на pnpm; **один Node-процес** обслуговує API, статику веб-панелі, Telegram-бота (long-polling) і cron. Деталі — у `CLAUDE.md` та `PROJECT_MAP.md`.

## Як швидко увійти в проєкт

1. Прочитай `CLAUDE.md` (5 хв) — отримаєш модель системи й правила.
2. За потреби глибше — `PROJECT_MAP.md` (карта + маршрути + ризики).
3. Для конкретної ділянки — README відповідного пакета.
4. Перевір середовище за потреби: `pnpm run typecheck`. Прод-процес: `pm2 list` / `pm2 logs grafik-bot`.

Типовий цикл змін: правка коду → `pnpm run typecheck` → (для бекенду) `pnpm --filter @workspace/api-server run build` → `pm2 restart grafik-bot`. Зміна схеми → SQL через `psql` + `pnpm run typecheck:libs`.

## Правило: не сканувати весь репозиторій без потреби

`bot/index.ts` (~3200 рядків) і `routes/admin-api.ts` (~2500) — великі; повне читання дороге й рідко потрібне.

- **Спершу** звіряйся з `PROJECT_MAP.md` (маршрути/модулі) і README — знайди точне місце.
- Читай **цільово**: `grep`/пошук по сим"волу, потім читання потрібного діапазону рядків, а не файлу цілком.
- `node_modules/`, `dist/`, `*.tsbuildinfo`, `pnpm-lock.yaml`, `artifacts/mockup-sandbox/` — **не читати** без явної причини.
- `lib/api-spec|api-zod|api-client-react` — скафолд, майже не використовується; не вивчати без потреби.
- Не запускай масові рекурсивні `grep`/`ls -R` по всьому репо «про всяк випадок».

## Як формулювати майбутні задачі

Щоб робота йшла швидко й точно, у задачі бажано вказати:

1. **Шар(и):** бот / API / веб / схема БД / cron — і конкретний файл, якщо відомий.
2. **Очікувана поведінка** + критерій готовності (як перевірити).
3. **Дані/ролі:** кого стосується (owner/scheduler/driver, працівник/водій), чи фінансові поля (owner-only).
4. **i18n:** чи є нові тексти (тоді потрібен EN-переклад).
5. **Схема:** чи потрібні зміни таблиць (тоді — SQL через psql + `typecheck:libs`).
6. **Перевірка:** typecheck/build, за потреби `pm2 restart` і смоук-тест.

Приклад добре сформульованої задачі: «У `admin-api.ts` додати поле X у `GET /workers` і показати його колонкою у `web/src/pages/Workers.tsx`; нове поле зі схеми Y; додати EN-переклад заголовка; перевір typecheck».

## Known risks / TODO

**Ризики (повний перелік — у `PROJECT_MAP.md → Ризикові місця`):**
- один polling-інстанс бота (інакше 409);
- немає міграційних файлів — схема й БД синхронізуються вручну (psql);
- один процес: падіння кладе все (рятує pm2 autorestart);
- Tailwind v4 — лише буквальні класи; i18n-дублі → TS1117; фінанси owner-only;
- секрети не комітити; Google Sheet/Drive мають бути розшарені на service account.

**Можливі TODO / напрямки (підтверджувати з власником перед роботою):**
- режим «всіх» (`genMode='all'`) генерує **Пн–Сб** — за потреби зробити налаштовуваним (Пн–Пт / +Нд);
- частина глибоких екранів офіс/водій-бота лишається лише українською (меню/привітання — двомовні);
- завантаження файлів документів — наразі лише посилання (`fileUrl`), без прямого аплоуду;
- `bot/index.ts` і `admin-api.ts` великі — за потреби обережно дробити на модулі.

> Перед руйнівними діями (видалення даних, масові зміни) — підтверджуй обсяг. Це активна система (живий бот, реальні дані).
