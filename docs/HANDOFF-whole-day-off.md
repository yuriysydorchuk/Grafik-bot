# Handoff — вихідний на цілий день (без призначеної зміни)

Завершено й **задеплоєно на прод**. Коміти: `0ff5ca5` (фіча) + `effa0af` (текст кнопки). Сесія 2026-07-05. SQL накатано локально і на проді.

---

## Контекст

Графікова часто складає графік лише на день наперед, тож працівник не міг відпроситися заздалегідь — бот пропонував вихідний тільки на **призначені зміни** затвердженого графіку. Тепер вихідний можна попросити трьома шляхами; всі рішення — за графіковою.

## Що зроблено

### Схема
`absence_requests.shift` став **nullable**: `NULL` = вихідний на цілий день.
```sql
ALTER TABLE absence_requests ALTER COLUMN shift DROP NOT NULL;  -- накатано (лок + прод)
```

### Бот — «🏖 Взяти вихідний» (`bot/index.ts`, 5 мов)
Одне меню з трьох джерел:
1. **призначені зміни** затверджених тижнів (≥24 год) — як раніше, запит на конкретну зміну;
2. **дні із заповненою диспозицією** (цей + наступний тиждень, без призначеної зміни) — «📋 дата — попросити вихідний на цей день»;
3. **«📅 Вихідний на інший день»** → пікер наступних **14 днів** (від завтра).

Пункти 2–3 → запит з `shift = NULL`. Гард: один активний (не-rejected) запит на дату (`absTakenDates()`); хелпери `absYmd`/`absDateOf` там же. Нові callback'и: `absother`, `absday:<YYYY-MM-DD>`.

### Розгляд (бот + веб)
- Цілоденний запит адмінам: «🏖 Запит на вихідний (цілий день)» — лише **Підтвердити / Відхилити** (заміна прив'язана до конкретної зміни, для цілоденних її нема; `POST /absence-requests/:id/substitute` повертає 400, invite-callback'и мають guard).
- Approve цілоденного (бот `absence_approve_*` і веб `/absence-requests/:id/approve`): всі **вже призначені** scheduled-зміни того дня → `absent`.
- Відповідь працівнику його мовою: `notif.absAcceptedDay` / `notif.absRejectedDay` (5 мов); бот-approve тепер теж шле мовою працівника (раніше хардкод uk).

### Генератор (`scheduleGenerator.ts`)
`isAbsent(wid, day, shift)` = зміна-специфічний сет **або** цілоденний `absentDaySet` (запити з `shift NULL`, статус ≠ rejected) — людина з активним запитом не потрапляє в жодну зміну дня. Перевірка у всіх 3 місцях добору.

### ⚠️ Маркери
- **Графік** (`GET /schedule`): цілоденний запит розгортається в `absenceByWorker` на всі зміни 1..6 (`??=` — конкретніший запит має пріоритет). Schedule.tsx уже вмів показувати «Відпрошується (не підтверджено)» — веб не чіпали.
- **Доступність** (`GET /availability` + `Availability.tsx`): нове поле `dayOff: Record<day, status>` на рядку працівника → ⚠️ (pending) / 🏖 (accepted) у клітинці дня. Працівник із запитом **без** рядків диспозиції додається в список синтетичним рядком (`source: "dayoff"`), щоб графікова бачила запит під час складання.
- **Absences.tsx**: «🏖 цілий день» замість номера зміни; блок замін схований для цілоденних.

## Що знати наступній сесії

- **`absence_requests.shift` тепер `Shift | null`** — будь-який новий код по цій таблиці мусить обробляти NULL (TS підкаже; `SHIFT_SHORT[r.shift]` без guard'а — краш).
- Цілоденний запит **блокує генерацію вже в статусі pending** (як і зміна-специфічний — семантика «не rejected» існувала до фічі, не змінювали).
- Пікер «інший день» — горизонт 14 днів від завтра; правило 24h діє лише для запитів на призначену зміну.
- `GET /availability`: рядки із `source: "dayoff"` мають порожні `days` — фронтові агрегації (`buildSummary`) їх безболісно ігнорують.
- Локальний тест-бот: після правок треба `build` + `pm2 restart grafik-bot` (локальний pm2 тримає старий dist у пам'яті — «нічого не змінилося» = забув рестартнути).
- Паралельно в репо з'явився `HANDOFF-mileage-edit.md` (корінь) від іншої сесії — не з цієї фічі.
