# Handoff — позапланові від водія: fuzzy-матчинг у боті + привʼязка у веб-графіку

Завершено й **задеплоєно на прод**. Коміт: `3e7c2a3`. Сесія 2026-07-05. Схема БД не мінялася (`unplanned_workers.worker_id` вже був nullable — SQL не потрібен).

## Контекст

Водій додає людину в зміну вручну у двох місцях бота: «✅ Посадка → ➕ Додати людину» (`boarding:add_name`) і меню «➕ Позаплановий працівник» (`unplanned:enter_name`). Матчинг був примітивний (точний код або перший `includes` по імені): «Iwan Kowalski» проти «Kowalski Iwan», кирилиця чи одрукування — і запис лишався в `unplanned_workers` голим текстом (`worker_id NULL`) **назавжди**: без явки, без годин, без способу привʼязати потім.

## Що зроблено

### Бот — `bot/workerMatch.ts` (новий модуль, юніт-тести поруч)

- `normalizeName`: польські діакритики (`ł→l`, `ż→z`…), транслітерація кирилиці (укр+рос), фолдинг взаємозамінних літер (`w→v`, `y/j→i`, `g→h`). Діграфи (ж/rz, ш/sz) добирає потокенний Levenshtein.
- `matchWorker`: точний `workerCode` → одразу впевнений; інакше скоринг по токенах **незалежно від порядку слів** (exact/prefix/1–2 одрукування). Три результати:
  - **confident** — авто-привʼязка (поріг 0.75 для ≥2 слів, 0.85 для одного; потрібен відрив ≥0.2 від другого кандидата);
  - **candidates** (до 4) — водієві inline-кнопки «Знайшов у базі схожих. Хто це?» + «➕ Ні, додати як введено» (actions `unpk:`/`brdpk:`, стани `unplanned:pick`/`boarding:add_pick`);
  - нічого — вільний текст, як раніше.
- Підключено в **обидва** флоу (`bot/index.ts`); спільні хелпери `candidatePickKb`, `saveUnplannedWorker`, `boardAddPerson`. Сповіщення адмінам про «не в базі» тепер додає ймовірних кандидатів. Вільний текст у Markdown — через `mdSafe` (раніше не було).

### API + веб — привʼязка графіковою (fallback)

- `GET /schedule`: `unplanned[key]` тепер `{ id, name, workerId }` (було лише `name`).
- **`POST /unplanned/:id/link { workerId }`** (гейт `editData`, `admin-api.ts` поруч зі schedule-роутами): проставляє `worker_id` + канонічне імʼя і **створює `schedule_entries` (status=present, pickedUpBy=водій)**, якщо такого entry ще немає — години зараховуються, як при посадці.
- `Schedule.tsx`: непривʼязані extras мають жовту кнопку «прив'язати» → модалка з пошуком по **всіх** активних працівниках (людина може бути з іншої фабрики); привʼязані — бейдж «➕ додатковий», як раніше.
- i18n: EN+RU для нових рядків бота (водійський флоу), EN для веба.

## Що знати наступній сесії

- **Не додавати нових `includes`-матчів по імені** — будь-який матчинг введеного водієм/людиною імені проти бази йде через `matchWorker` з `bot/workerMatch.ts` (правило додано в CLAUDE.md).
- Привʼязка через веб **створює явку present** — свідоме рішення (дзеркалить посадку): людина фізично була на зміні. Якщо entry на цю пару worker+week+day+shift+factory вже існує — другий не створюється.
- Водій, що натиснув «Ні, додати як введено», явно відхилив кандидатів — у цьому випадку адмінам підказки не шлються (шлються лише коли матчер не знайшов нічого впевненого і водія не питали).
- У боті лишився дубляж: матчений при посадці працівник має і entry (present), і рядок в `unplanned_workers` → у веб-явці видно і рядок явки, і бейдж «додатковий». Це **існуюча** поведінка, не регресія.
- Пороги матчера (0.55 кандидат / 0.75–0.85 впевнений / відрив 0.2) — константи у `workerMatch.ts`; якщо водії скаржитимуться на зайві питання або хибні авто-привʼязки — крутити там і додавати кейс у `workerMatch.test.ts`.
