# Handoff: редагування всіх даних працівника з профілю

Дата: 2026-06-18 · Статус: **готово, не задеплоєно на прод**

Раніше сторінка профілю працівника (`/workers/:id`) була **тільки для читання** —
редагувати дані можна було лише зі списку Workers. Тепер у профілі є кнопка
**«Редагувати»**, яка відкриває повну форму з усіма полями.

---

## Що зроблено

- У шапку профілю додано кнопку **«Редагувати»** → відкриває форму редагування всіх даних.
- Щоб не дублювати форму, наявну `WorkerModal` **винесено зі сторінки Workers у спільний
  компонент** `components/WorkerModal.tsx` і перевикористано в обох місцях (список + профіль).
- Розширено набір редагованих полів до «всіх»:
  - **Код працівника** — раніше редагувався лише при створенні; тепер і при редагуванні
    (валідація «лише цифри» + перевірка унікальності, виключаючи самого працівника).
  - **Мова бота** — нове поле вибору (uk / en / es / ru / pl).
  - Решта вже були: ім'я, фірма, фабрика, посада, стать, закріплена зміна, Telegram ID,
    фінанси (ставка / студент / до 26 — лише owner).

---

## Файли змінено

**Бекенд** (`artifacts/api-server/`)
- `src/routes/admin-api.ts` — `PATCH /workers/:id` тепер приймає `workerCode`
  (валідація `^\d+$` + дуб-чек, що виключає поточного працівника; порожнє → `null`)
  та `language` (порожнє → `null`). Решта полів приймалися й раніше.

**Веб** (`artifacts/web/`)
- `src/components/WorkerModal.tsx` — **новий** спільний компонент. Винесено зі `pages/Workers.tsx`;
  додано: завжди редаговане поле «Код», select «Мова бота» (константа `LANGUAGES`).
  Тепер і create, і edit шлють одне тіло `base` (POST сам авто-генерує код, якщо порожній).
- `src/pages/Workers.tsx` — імпортує `WorkerModal` з компонента; локальне визначення
  видалено; прибрано невикористані імпорти (`patch`, `Label`, `Modal`).
- `src/pages/WorkerDetail.tsx` — кнопка «Редагувати» в шапці; додано запити
  `factories` / `companies` + `useMe` (isOwner); стан `editing`; будує `Worker`-обʼєкт
  `workerForEdit` з профілю для модалки; після збереження інвалідує
  `["worker", id]` і `["workers"]`.
- `src/lib/api.ts` — у тип `Worker` додано `language?: string | null`.
- `src/lib/i18n.tsx` — нові ключі: «Код (лише цифри)», «Мова бота» (+EN).

## Що потрібно знати наступній сесії

### Деплой на прод (ще не зроблено)
**Міграція БД НЕ потрібна** — колонки `worker_code` і `language` вже існують у схемі
(`lib/db/src/schema/workers.ts`). Звичайний цикл:
```bash
cd ~/grafik-bot && git pull
bash deploy/build.sh
pm2 restart grafik-bot && pm2 save
```

### Перевірено
- `pnpm run typecheck` ✅, build веб ✅, build бекенду ✅, дублів i18n немає (`uniq -d`).
- **Локальний запуск сервера НЕ робили** — другий polling-інстанс бота дав би `409 Conflict`
  з продом. Ручний UI-тест профілю → «Редагувати» лишається за бажанням на тестовому стенді.

### Дрібниці / можливі наступні кроки
- Старий i18n-ключ `"Код (необов'язково, лише цифри)"` лишився в словнику, але більше не
  використовується (безпечно; за бажанням прибрати).
- `workerCode` тепер можна **очистити** (стане `null`). PG unique допускає кілька null —
  конфлікту немає. Якщо код десь критичний для бізнес-логіки — варто заборонити очищення.
- Редагування мови адміном перетирає вибір, який працівник робить у боті — поведінка очікувана
  (поле «всіх даних»), але майте на увазі.
- Статус active/fired редагується окремо кнопками «Звільнити»/«Відновити» (не в цій формі).
