# Handoff — рапорти/години/історія по фабриках (переведення в межах місяця) + права головного водія на вебі

Завершено й **задеплоєно на прод**. Коміти: `84cbb31` (per-factory рапорти/години/історія) + `e8c5866` (головний водій керує водіями на сайті). Прод HEAD `e8c5866`.

---

## Контекст / рішення власника

Працівники **не працюють на двох фабриках одночасно** — але їх можуть **перевести** з однієї на іншу посеред місяця (або звільнити з однієї і найняти на іншу). Тому:
- рапорт за місяць може бути **на кожну фабрику окремо** (раніше другий рапорт перезаписував перший);
- в Обліку годин людина в місяць переведення має бути видима **в обох фабриках** зі своїми годинами;
- у профілі — історія по кожній фабриці; сам профіль (код, документи, Telegram) при переведенні/перенаймі не змінюється.
- `workers.factoryId` лишився **одним полем** («поточна фабрика»); переведення = зміна фабрики в редагуванні працівника. Від ідеї множинної прив'язки (join-таблиця, зміни генератора) власник відмовився свідомо.

## Що зроблено

### Схема (`lib/db/src/schema/workers.ts`) — SQL накатано на прод
`monthly_reports`: unique **(worker_id, month)** замінено на:
- `monthly_reports_worker_month_factory_uniq` — UNIQUE **(worker_id, month, factory_id)**;
- `monthly_reports_worker_month_nofactory_uniq` — частковий UNIQUE (worker_id, month) `WHERE factory_id IS NULL` (легасі/ручні записи без фабрики — максимум один).

```sql
DROP INDEX IF EXISTS monthly_reports_worker_month_uniq;
CREATE UNIQUE INDEX IF NOT EXISTS monthly_reports_worker_month_factory_uniq ON monthly_reports (worker_id, month, factory_id);
CREATE UNIQUE INDEX IF NOT EXISTS monthly_reports_worker_month_nofactory_uniq ON monthly_reports (worker_id, month) WHERE factory_id IS NULL;
```

### Бот (`bot/index.ts`)
- Список фабрик при подачі рапорту = фабрики із **затверджених змін звітного місяця** (тижні беруться з запасом −6 днів, кожна зміна фільтрується за фактичною датою) **+ поточна фабрика** працівника як fallback. Раніше брались усі фабрики за весь час.
- `submitMonthlyReport`: upsert по **(workerId, month, factoryId)**; якщо фабрику не вдалось визначити (factoryId null) — ручний select→update/insert, бо частковий індекс не годиться як onConflict target.

### API (`routes/admin-api.ts`)
- **`GET /hours`** — рядки тепер ключуються **(workerId, factoryId)** (`rowKey = "id|facId"`): у місяць переведення працівник з'являється в обох групах-фабриках зі своїми змінами/годинами. Рапорт-години метчаться по (worker, factory); легасі-рапорти без фабрики чіпляються до рядка **поточної** фабрики працівника (else — першого рядка). Fallback «усі активні без змін» — лише якщо у працівника немає жодного рядка.
- **`POST /hours/report`** — приймає `factoryId` (з групи на сторінці Hours); без нього — фабрика працівника. Очищення/upsert — по (worker, month, factory), null-safe.
- **`GET /workers/:id`** — нове поле **`factoryHistory[]`**: `{factoryId, factoryName, shifts, hours, absent, firstDate, lastDate}` по кожній фабриці зі `schedule_entries` (всі статуси рахуються в період, present → зміни/години, absent → невиходи). Сортування за lastDate desc.
- **`POST/PATCH/DELETE /drivers`** — тепер під **`requireAnyCap("editData", "assignDrivers")`** (новий хелпер у `lib/auth.ts`). Фікс: головний водій (веб-роль `driver`, caps `["assignDrivers"]`) отримував 403 «forbidden» при додаванні водія на сайті, хоча через бота міг. Призначення головного водія всередині PATCH — як і раніше лише `isMain`.

### `services/drive.ts`
`buildReportHoursExcel` — рядок на кожну пару (працівник, фабрика з рапорту); працівник без рапорту — один рядок під поточною фабрикою. Фільтр `?factoryId=` враховує фабрику рапорту, а не лише поточну.

### Веб
- `pages/Hours.tsx`: ключ рядка `workerId-factoryId`; ручне вписування годин шле `factoryId` групи.
- `pages/WorkerDetail.tsx`: блок **«Історія по фабриках»** (Фабрика · Період · Зміни · Години · Невиходи, бейдж «поточна») між KPI та документами; ховається, якщо історії немає.
- `lib/i18n.tsx`: EN для «Історія по фабриках» / «Період» / «поточна».

## Що знати наступній сесії

- **Звільнення → інша фабрика** нових механізмів не потребує: «звільнити» зберігає запис (`status="fired"`), «відновити» + зміна фабрики = переведення; історія по старій фабриці видима через `factoryHistory` (вона живе у `schedule_entries`, які зберігають фабрику кожної зміни).
- **`POST /hours/report-remind` не чіпали**: працівник вважається «здав», якщо є **хоч один** рапорт за місяць — у місяць переведення про другу фабрику не нагадає. Свідомо лишено; за потреби — перевіряти по кожній фабриці змін місяця.
- `report:select_month` у `bot/index.ts` (~3342) — **мертвий стан** (ніде не `setState`-иться), лишений як є.
- Дубль-чек i18n-ключів `uniq -d` по всьому файлу тепер **фальшиво спрацьовує**: у файлі два словники (uk→en і частковий uk→ru), ключі легально повторюються. Реальні дублі в межах словника ловить `tsc` (TS1117).
- Легасі-рапорти без `factory_id` в БД лишились як є — API прив'язує їх до поточної фабрики працівника на льоту; міграцію даних не робили.
- Деплой: `ssh grafik` → `git pull` → SQL (див. вище, вже накатано) → `bash deploy/build.sh`. Перевірено: healthz ok, бот polling, індекси в `pg_indexes` на місці.
