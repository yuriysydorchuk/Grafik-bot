# ALERTING — моніторинг помилок і Telegram-алерти

> Як прод повідомляє про збої. Експлуатація — [RUNBOOK.md](RUNBOOK.md), середовище — [PRODUCTION.md](PRODUCTION.md).

Мета: якщо щось ляже або працює неправильно — owner швидко отримує **коротке** повідомлення в Telegram.

---

## Як це влаштовано

- Ядро: [`artifacts/api-server/src/lib/alerts.ts`](../../artifacts/api-server/src/lib/alerts.ts) — `sendAlert(...)`.
- Алерти йдуть **тим самим ботом** (лише вихідний `sendMessage`) — окремого polling-інстансу немає,
  правило «один інстанс бота» не порушується.
- **Best-effort:** надсилання алерту ніколи не кидає помилку назад у запит/бота/cron. Усе, що алертиться,
  паралельно пишеться в pino-логи (pm2), навіть коли алерти вимкнені.

### Що моніториться
| Джерело | Що ловить |
|---|---|
| `api` | необроблені помилки HTTP-запитів (Express error-middleware) |
| `process` | `unhandledRejection`, `uncaughtException` (фатальні — процес перезапускається pm2) |
| `bot` | помилки обробників Telegraf (`bot.catch`) |
| `cron` | падіння планових job (`services/scheduler.ts`) |
| `startup` | старт/рестарт сервісу (лише якщо алерти ввімкнені) |

### Healthcheck
`GET /api/healthz` повертає `{ status, db, bot, uptimeSec }`:
- `db` — швидкий `select 1` (`ok`/`down`);
- `bot` — чи стартував polling (`botLaunched`);
- HTTP **200** якщо здоровий, **503** якщо БД недоступна.
- Публічний ендпоінт — **без stack trace, env, токенів і внутрішніх текстів помилок БД**.

## Налаштування (env)

| Змінна | Дефолт | Призначення |
|---|---|---|
| `ALERTS_ENABLED` | `false` | `true` — вмикає надсилання алертів (інакше лише логування) |
| `ALERT_TELEGRAM_CHAT_ID` | — | **основний** одержувач. Якщо порожній — фолбек на `is_main` адміна |
| `ALERT_COOLDOWN_SECONDS` | `300` | пер-помилковий cooldown (антиспам) |

Увімкнути на сервері:
```bash
cd /root/grafik-bot
nano .env          # ALERTS_ENABLED=true, ALERT_TELEGRAM_CHAT_ID=<твій chat_id>
pm2 restart grafik-bot --update-env
```

### Як дізнатися свій `chat_id`
1. Напиши боту будь-що в Telegram.
2. Відкрий (зі своєї машини, токен не світи стороннім):
   `https://api.telegram.org/bot<TOKEN>/getUpdates`
3. Візьми `result[].message.chat.id` (для приватного чату — додатнє число; для групи — від'ємне).

> Фолбек на `is_main` адміна спрацьовує лише якщо `ALERT_TELEGRAM_CHAT_ID` порожній, читається лінива
> з БД і ніколи не ламає запуск. Рекомендовано все одно задати `ALERT_TELEGRAM_CHAT_ID` явно.

## Антиспам (захист від лавини алертів)

- **Cooldown за ключем:** однакова помилка не частіше, ніж раз на `ALERT_COOLDOWN_SECONDS`.
- **Дедуплікація:** ключ = `service|kind|source|normalized(message)` (числа/id в повідомленні
  замінюються на `#`, тож однотипні помилки злипаються).
- **Глобальний ліміт:** не більше ~10 алертів / 5 хв; зайве придушується.
- Придушені однотипні рахуються й показуються у наступному алерті: `(+N similar suppressed)`.
- Стан — у пам'яті процесу; після рестарту скидається (це нормально).

## Формат алерту (короткий)
```
🚨 grafik-bot · api
TypeError @ POST /api/workers
Cannot read properties of undefined (reading 'id')
🕒 19.06.2026, 14:32:05
```
Лише: сервіс, тип помилки, шлях/джоба, sanitized-повідомлення, час (Europe/Warsaw).

### Що НІКОЛИ не потрапляє в алерти/логи
Токени, паролі, `SESSION_SECRET`, connection-string БД, повний `.env`, тіло/query/headers запитів,
імена/телефони/дані працівників і кандидатів. Повідомлення додатково проганяється через
санітайзер (редакт токеноподібних рядків, БД-URL, довгих секретів) і обрізається до ~300 символів.

## Обмеження ⚠️

Алерти шле сам бот — тож якщо **впав увесь процес або сервер**, самоалерт не надійде (нікому слати).
Часткове падіння (помилка запиту/джоби/апдейту, недоступна БД) — покривається; повне — ні.
`uncaughtException` намагається відправити алерт перед виходом, але гарантії немає.

**Рекомендація (поза кодом, налаштовує owner):** зовнішній uptime-монітор, що пінгує
`https://161.97.117.151.sslip.io/api/healthz` кожні 1–5 хв і шле сповіщення при недоступності/`503`:
- [healthchecks.io](https://healthchecks.io), [UptimeRobot](https://uptimerobot.com),
  [Better Stack](https://betterstack.com) — безкоштовні тарифи достатні.
- Налаштувати алерт у Telegram/email монітора. Це єдиний спосіб дізнатися про повне падіння сервера.

## Перевірка / експлуатація

- **Тест:** з `ALERTS_ENABLED=true` після `pm2 restart` має прийти startup-алерт. Для тесту помилки —
  тимчасово смикнути неіснуючий API-роут, що кидає, і глянути алерт + лог.
- **Заглушити алерти:** `ALERTS_ENABLED=false` у `.env` → `pm2 restart grafik-bot --update-env`
  (логування лишається).
- **Логи:** усе, що алертиться, є в `pm2 logs grafik-bot` з полем `alert` — навіть коли алерти вимкнені.
- Дії при конкретних збоях — [RUNBOOK.md](RUNBOOK.md).
